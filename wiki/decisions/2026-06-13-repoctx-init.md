# repoctx init — meta-hook orchestrator design

**Date**: 2026-06-13. **Issue**: `57f88d2`. **Epic**: `40c8baa`. **Milestone**: v0.6.0.

## Summary (CHANGELOG-worthy)

`repoctx init` becomes the first-class onboarding command. It installs a
**committed, version-marked shell script** (`.repoctx/hook.sh`) as the
sole `PreToolUse → Bash` entry and points Claude Code's settings at it.
The script is a *dumb pipe* — no `jq`, no JSON parsing — that execs
`repoctx hook claude`, where all rewrite/JSON/chain logic lives. repoctx
becomes the meta-hook: when [rtk](https://github.com/rtk-ai/rtk) is
present it is **chained underneath** (no degradation), and any *other*
tool's hook is detected and refused (it would race). `init -g` does the
same at user-global scope, displacing a pre-existing rtk hook with an
explicit prompt. A teammate who clones the repo without `repoctx`
installed gets a clear "install repoctx" message instead of a silent
no-op, because the bootstrap lives in a committed file they can read.

This doc pins the design; children `e06f463`, `89bfb29`, `4b2af2a`,
`15f5170`, `b2ad123`, `cba3b50`, `2307c32`, `42ec6e9`, `43142e1`,
`ec698bb`, `0a338d7`, `ed3e222`, `ad58ede` implement against it.

## Problem

Three problems, one design:

1. **Project-local UX gap.** `rtk init` (local) writes CLAUDE.md but
   *not* a hook — local-only rtk users get no token compression.
2. **Global-vs-local race.** Claude Code merges `PreToolUse` hooks
   across user-global + project scopes and runs same-matcher hooks in
   **parallel**, with `updatedInput` resolved non-deterministically
   (last writer wins). Global rtk + local repoctx both rewrite
   `rg`/`grep` → corrupted, run-to-run-variable commands.
3. **Missing-binary opacity.** A bare `repoctx hook claude` entry on a
   teammate's clone with no `repoctx` on PATH fails silently; the user
   has no idea why "the hook does nothing".

## Claude Code execution model (verified)

Carried forward from
[`2026-06-12-rewrite-hook-design.md`](2026-06-12-rewrite-hook-design.md),
verified against Claude Code 2.1.112 `cli.js`:

- Multiple `PreToolUse` hooks under the same matcher run **in parallel**
  (`Array.map` + `Promise.race`); no `priority`/`order` field exists.
- `updatedInput` has **no precedence** — last hook to finish wins,
  non-deterministic.
- Hooks are **merged across scopes** (user-global `~/.claude/settings.json`
  + project `.claude/settings.json` + `.claude/settings.local.json`).
- Claude Code **prompts the user to approve project-defined hooks** before
  first execution (its built-in trust gate for untrusted repos).

**Consequence**: two independently-registered Bash hooks cannot coexist
deterministically. The only correct model is **a single entry**, owned by
repoctx, that chains anything else *in-process*.

## DECIDED — dumb-pipe script (no jq)

The committed script does **not** parse JSON. All JSON + rewrite + chain
logic stays in the Rust binary, where it is unit-tested. Script
responsibilities, complete:

1. Resolve `repoctx`. On miss: print an install link to stderr; if
   `RTK_CHAIN=1` and `rtk` is present, `exec rtk hook claude` (preserve
   rtk's savings even without repoctx); else `exit 0` (passthrough).
2. Version guard, cached at `$XDG_CACHE_HOME/repoctx-hook-version-ok`
   (mirrors rtk's once-and-cache pattern).
3. `exec repoctx hook claude --rtk-chain=$RTK_CHAIN` — stdin untouched.

**Rationale.** rtk's script hard-depends on `jq` and dies silently when
it's absent — a real install-failure mode. We already ship a
JSON-speaking binary; the script is only a bootstrap + a human-readable
visibility layer. `RTK_CHAIN` stays a plain script variable so a user can
read and flip it by editing the committed file.

**Consequences.**
- `repoctx hook claude` gains `--rtk-chain=0|1`. Chain execution
  (spawning `rtk hook claude` with the original stdin on passthrough)
  moves *into* the binary — testable in Rust, no shell loop.
- The `hook.chain_commands` config key + its shell-loop dispatch are
  retired (see Migration). Chaining is now rtk-specific and in-binary.
- `repoctx rewrite` (`e06f463`) is a standalone debug/bench utility, not
  a script dependency.

### Final template — `crates/repoctx/assets/hook.sh.tmpl`

`include_str!`'d into the binary; rendered by `init`/`doctor` with the
three `{{...}}` variables substituted. Authoritative form:

```bash
#!/usr/bin/env bash
# repoctx-hook-version: 1
# Generated by `repoctx init` — safe to read; edit RTK_CHAIN below to
# toggle rtk chaining. All rewrite/JSON/chain logic lives in the binary.
set -euo pipefail

RTK_CHAIN={{RTK_CHAIN}}          # 0 | 1  — chain rtk underneath on passthrough
MIN_VERSION="{{MIN_VERSION}}"    # binary version that generated this script
REPOCTX="{{REPOCTX_BIN}}"        # "repoctx" (PATH) or an absolute path

if ! command -v "$REPOCTX" >/dev/null 2>&1; then
  echo "[repoctx] not installed — install: cargo install repoctx" >&2
  echo "          (or: nix run github:mikolajmikolajczyk/repoctx)" >&2
  if [ "$RTK_CHAIN" = "1" ] && command -v rtk >/dev/null 2>&1; then
    exec rtk hook claude            # keep rtk's savings even without repoctx
  fi
  exit 0                            # passthrough; never block the Bash tool
fi

# Version guard, cached so we don't re-spawn on every hook call.
CACHE_DIR="${XDG_CACHE_HOME:-$HOME/.cache}"
if [ ! -f "$CACHE_DIR/repoctx-hook-version-ok" ]; then
  if ! "$REPOCTX" hook claude --supports-rtk-chain >/dev/null 2>&1; then
    echo "[repoctx] $REPOCTX is older than $MIN_VERSION; upgrade for hook chaining" >&2
  fi
  mkdir -p "$CACHE_DIR" 2>/dev/null || true
  touch "$CACHE_DIR/repoctx-hook-version-ok" 2>/dev/null || true
fi

exec "$REPOCTX" hook claude --rtk-chain="$RTK_CHAIN"
```

`--supports-rtk-chain` is a trivial probe subcommand that exits 0 on any
binary new enough to honor `--rtk-chain` (it's a no-op flag check); older
binaries exit non-zero on the unknown flag, so the guard degrades to a
one-line warning, never a hard failure. The cache is cleared by `doctor`.

### `repoctx hook claude --rtk-chain` behavior

- Rewrite rule matches → emit the `updatedInput` JSON, exit 0. rtk is
  **not** consulted (the answer is already structured).
- Passthrough + `--rtk-chain=1` + rtk on PATH → spawn `rtk hook claude`,
  feed it the original stdin, forward its stdout + exit code verbatim.
- Passthrough + `--rtk-chain=1` + rtk absent → warn once (sentinel
  `$XDG_CACHE_HOME/repoctx-rtk-missing-warned`), then passthrough.
- Passthrough + `--rtk-chain=0` → exit 1 (silent passthrough).
- Flag resolution when invoked directly (no flag): `--rtk-chain` >
  `hook.use_rtk` config > default `0`.

## DECIDED — CLI surface

```
repoctx init [-g] [--agent claude] [--rtk auto|on|off] [--yes] [--force] [--dry-run]
repoctx init --uninstall [-g] [--yes] [--force] [--dry-run] [--restore-backup]
repoctx hook claude [--rtk-chain=0|1]      # JSON stdin/stdout handler
repoctx hook doctor [-g] [--fix]           # drift/tamper check + regen
repoctx rewrite <cmd>                       # debug/bench utility, exit-code protocol
repoctx hook install <agent>                # low-level escape hatch (NOT deprecated in v0.6.0)
```

`init` is the supported entry point; `hook install` stays for scripting
and as the primitive `init` is built on. No deprecation marker in v0.6.0.

### `repoctx rewrite` exit-code protocol (`e06f463`)

`0` = rewrite found (stdout = rewritten command), `1` = passthrough, `2`
= deny (future), `3` = ask (future). Stdout only on 0/3; diagnostics to
stderr. Standalone — wraps the same pure decision function the corpus
(`573eccc`) already tests; not used by the script.

## DECIDED — scope write-targets

| Command | settings file | script path |
|---|---|---|
| `repoctx init` | `<repo>/.claude/settings.json` | `<repo>/.repoctx/hook.sh` |
| `repoctx init -g` | `~/.claude/settings.json` | `~/.claude/repoctx-hook.sh` |

Each writes a single `PreToolUse → Bash` entry whose `command` is the
exact script path (never a shell expression). Because Claude Code merges
scopes, installing at *both* scopes double-fires — the race ruleset below
refuses that.

## DECIDED — race-detection ruleset

`init` scans every `PreToolUse → Bash` entry across all three scopes and
refuses (exit non-zero) when an install would create a race, printing the
offending entry + actionable options. `--force` overrides every row.

| Existing | Target install | Behavior |
|---|---|---|
| Foreign hook, any scope | any | refuse + table + options; `--force` overrides |
| global rtk | local | refuse; offer: `init -g` (recommended) / uninstall global rtk / `--force` |
| global repoctx | local | refuse (double-fire); offer `doctor` refresh of global / `--force` |
| local repoctx | global | refuse (double-fire); offer remove local first / `--force` |
| same-scope repoctx | same scope | idempotent doctor-mode refresh, no prompt |

## DECIDED — hook classification

Allowlist = `{repoctx, rtk}`. For each `PreToolUse → Bash` entry:

- Tokenize the `command`; take the first token.
- First token `repoctx` / `rtk` → that tool.
- First token is a path to a script → read its first ~8 lines, look for
  the marker `# <tool>-hook-version: <n>` (`42ec6e9`); classify by
  `<tool>`. No marker → **Foreign**.
- First token `bash -c '...'` (or any wrapper that hides the real
  program) → **Foreign** (conservative; we don't parse nested shells).
- Unresolvable path / unknown program → **Foreign**.

Foreign anywhere blocks install without `--force`. `doctor` reports the
same scan but only warns.

## DECIDED — trust model

The committed `.repoctx/hook.sh` is repo-controlled code the agent
executes; a malicious clone could ship a doctored script. Layers:

1. **Claude Code's own gate.** It prompts the user to approve
   project-defined hooks before first execution (verified, cli.js
   2.1.112). Approving a cloned repo's hooks is the user's trust
   decision — repoctx does not and cannot sandbox it.
2. **Exact-path invocation.** The settings entry runs `.repoctx/hook.sh`
   by exact relative path, never an interpolated/shell expression, so the
   attack surface is exactly that one reviewable file.
3. **Drift/tamper visibility via `doctor`.** `repoctx hook doctor`
   re-renders the expected script from the embedded template + current
   config + binary version, and byte-compares it to the on-disk file. A
   mismatch is a loud warning with a diff summary and an offer to
   regenerate (`--fix`). No sha256 side-file is needed — the binary can
   always re-derive the canonical content. This catches both tampering
   and staleness after an upgrade.

## DECIDED — config keys (final)

Add:
- `hook.use_rtk` = `auto` | `on` | `off` (default `auto`). At `init`
  time, `auto` = detect rtk (PATH + global-hook scan) and bake
  `RTK_CHAIN` into the script. At hook runtime (direct invocation),
  resolves the absent `--rtk-chain` flag.
- `hook.script_path` = read-only computed value (where `init` wrote the
  script); surfaced by `config show` + `doctor`. `config set` on it errors
  with a pointer to `repoctx init`.
- `hook.chainable` = `["rtk"]` — allowlist, newline-serialized. Structural
  future-proofing; only rtk is consulted in v0.6.0.

Remove:
- `hook.chain_commands` — chaining is in-binary + rtk-specific now
  (Migration handles existing rows).

Already gone (v0.5.3, `43aeaff`): `hook.ref`, `hook.no_cache`.

Kept: `hook.rewrite` (`auto`/`off`/`force`).

## DECIDED — migration v0.5.x → v0.6.0 (`43142e1`)

A v0.5.x install has an inline `repoctx hook claude` entry in
`.claude/settings.json` + a `hook.chain_commands` row. On the first
`repoctx init` (or `doctor`) under v0.6.0:

1. Detect the inline `repoctx hook claude` Bash entry → replace it with a
   `.repoctx/hook.sh` entry.
2. Read `hook.chain_commands`:
   - contains `rtk hook claude` → set `hook.use_rtk = on` and bake
     `RTK_CHAIN=1` into the generated script;
   - contains anything else → **warn**, print the value, and ask the user
     (do not silently drop a non-rtk chain).
3. Delete the `hook.chain_commands` row.
4. Generate `.repoctx/hook.sh` (+ `.gitattributes`, below).
5. Print a migration summary.

Detection signals: inline `repoctx hook claude` literal in settings;
presence of a `hook.chain_commands` row; absence of `.repoctx/hook.sh`.

## DECIDED — Windows policy (v1)

- Ship `.repoctx/hook.sh` (bash) only. Document the Git Bash requirement
  for Windows users.
- PowerShell variant (`hook.ps1`) is deferred to a follow-up.
- `init` writes/extends `.gitattributes` with `*.sh text eol=lf` so the
  committed script keeps LF endings and its executable bit survives
  checkout across platforms.

## DECIDED — prompt library

Hand-rolled minimal prompts (read a line from stdin, default on empty
input), **no new dependency**. Rationale: v0.5.3 just removed `ureq` +
`rustls` + `directories` to shrink the binary and drop the network
surface; adding `dialoguer` (~150 KB + its own deps) for three yes/no +
one single-select prompt is the wrong trade. The non-interactive path
(`--yes` + flags) is the primary one for agents/CI anyway; the
interactive path only needs simple line prompts. If arrow-key selects are
wanted later (multi-agent select), revisit `dialoguer` then.
